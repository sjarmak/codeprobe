## High

### [infra-lint-zfc] Allowlist path matching has a bidirectional suffix bug enabling false suppression

- File: `scripts/lint_zfc.py:191-193`
- What: `is_suppressed` checks three conditions: exact equality, `finding_posix.endswith("/" + entry_posix)` (finding is longer), and `entry_posix.endswith("/" + finding_posix)` (entry is longer). The third arm means a finding at `src/codeprobe/mining/extractor.py` is suppressed by an allowlist entry whose `file` is `evil/src/codeprobe/mining/extractor.py`. Verified in live test:
  ```
  finding  = src/codeprobe/mining/extractor.py  (line 82)
  entry    = evil/src/codeprobe/mining/extractor.py (lines 80-87)
  result   = suppressed: True   # WRONG
  ```
- Why it matters: The allowlist TOML is in-repo and its entries are maintained by reviewers, so an external attacker cannot inject it. The risk is internal: a misconfigured or overly broad entry path (e.g. during copy-paste) silently suppresses legitimate findings in unrelated source files that happen to share a basename/suffix.
- Suggested fix: Remove the third arm (`entry_posix.endswith("/" + finding_posix)`). Suppression should only fire when the finding path *ends with* the allowlist's `file` path — meaning the allowlist entry is the shorter suffix anchor, not the longer one. The first two conditions (exact equality and finding-is-longer) already cover all intended use cases.

### [p2-ci-pipeline] ZFC lint step is non-blocking (`continue-on-error: true`) — no enforcement

- File: `.github/workflows/ci.yml:56-58`
- What: The ZFC lint step runs but never fails the build. The comment says "A later work unit flips it to blocking." But this is not a ticket — it is a non-blocking check that self-documents it will be ignored indefinitely.
- Why it matters: CI non-blocking steps that flag violations create a boy-who-cried-wolf dynamic. Contributors stop reading the step output. When this is later flipped to blocking, it is likely to block unexpectedly because violations have accumulated silently. This is the opposite of the incremental ratchet strategy.
- Suggested fix: Either (a) flip to blocking now and confirm the allowlist covers all current violations (the full `src/codeprobe/mining/` scan shows zero findings even without `--allowlist`, so it appears safe to flip), or (b) track this as a bead with an explicit deadline and acceptance criterion "CI step `lint_zfc` exits non-zero on new violations."

### [p2-ci-pipeline] Coverage floor silently regressed from 80% to 70% without a bead

- File: `.github/workflows/ci.yml:53`
- What: The comment acknowledges the floor was lowered from the project-mandated 80% (`rules/common/testing.md`) to 70% "transitionally." There is no linked ticket, no deadline, and no ratchet mechanism.

### [r6-resolved-instruction] Deferred import of private function `_base_prompt` from production call site

- File: `src/codeprobe/cli/run_cmd.py:589`
- What: The R6 instruction resolution loop (inside the closure `_run_config`) imports `_base_prompt` — a private function (single-underscore prefix) — from `codeprobe.core.preamble`. This is a cross-module call on an implementation detail that may be refactored without notice.
- Why it matters: The function signature or existence of `_base_prompt` is not part of `preamble.py`'s public API. If it is renamed or absorbed into `compose_instruction` during a later simplification pass, the import breaks at runtime (not at import time, since it is a deferred import) — meaning the failure surfaces during `codeprobe run`, not when the module loads.
- Suggested fix: Either (a) expose a public `base_prompt(...)` in `codeprobe.core.preamble.__all__`, or (b) move the instruction-resolution logic into `core/preamble.py` as a thin helper `resolve_instruction_prompt(task_dir, exp_config, repo_root, ...)` so `run_cmd.py` never needs to compose the prompt itself.

## Medium

### [r5-trace-sqlite] `trace_recorder` typed as `object | None` in executor.py — loses type safety

- File: `src/codeprobe/core/executor.py:763`
- What: The parameter is annotated `trace_recorder: object | None = None`. The `set_trace_context` call is guarded by `callable(set_trace)` to avoid a hard import dependency. However, `TYPE_CHECKING` is already imported in `executor.py` (line 18) and used for other forward references — a `TYPE_CHECKING` guard around `from codeprobe.trace.recorder import TraceRecorder` and `trace_recorder: TraceRecorder | None = None` would restore proper type inference with zero runtime cost.
- Why it matters: `object | None` means mypy cannot verify the `record_event` call signatures on the recorder. Bugs in call-site argument names (e.g., `task_id` vs `config`) are silently accepted.
- Suggested fix: Use `TYPE_CHECKING`:
  ```python
  if TYPE_CHECKING:
      from codeprobe.trace.recorder import TraceRecorder

  def execute_config(
      ...,
      trace_recorder: "TraceRecorder | None" = None,
  ) -> ...:
  ```

### [r15-incremental-mining] `assert_tenant_owned` defined but never called in production paths

- File: `src/codeprobe/paths.py:134`, call sites: none
- What: `assert_tenant_owned` is exported in `__all__` and tested, but no production code in `src/codeprobe/` calls it on any read or write operation. The `MineState.open`, `cache purge`, and other tenant-scoped operations compute their own path from `tenant_root` and proceed without the ownership assertion. The security boundary the function was designed to enforce (cross-tenant access fails closed) is therefore advisory only.
- Why it matters: The boundary exists as a named contract but has no enforcement at runtime for any of the paths that matter. A caller that computes a path independently (e.g., via a future refactor that takes a path argument rather than tenant_id + repo_hash) can accidentally cross tenant boundaries with no guard.
- Suggested fix: Call `assert_tenant_owned` at each write boundary in `MineState.open` and in `cache_cmd.purge` before `rmtree`. It is a cheap resolved-path comparison. Add it as a habit rather than deferring to "callers will remember."

### [r_w-container-exec] Sandbox env vars passed as `KEY=VALUE` string — unvalidated key format

- File: `src/codeprobe/sandbox/runner.py:107`
- What: `argv += ["-e", f"{key}={value}"]` passes env vars to `docker run` without validating that `key` is a valid env-var name (no `=`, no whitespace, no null bytes). Docker's CLI accepts this as an argv token (no shell), so there is no shell injection. However, if `key` contains `=`, the argument becomes `A=B=C` which Docker parses as `A` → `B=C` — silently wrong. If `key` is empty, the result is `=val`, which Docker may refuse or accept unexpectedly.
- Why it matters: The `env` dict is caller-supplied. Current callers appear to use controlled key names, but no validation is enforced at the API boundary.
- Suggested fix: Add a validation guard before building the argv:
  ```python
  for key, value in (env or {}).items():
      if not key or "=" in key or "\n" in key:
          raise ValueError(f"Invalid env var key: {key!r}")
  ```

### [mining-writer-parity] False ZFC compliance claim in `assign_ground_truth_tiers` docstring

- File: `src/codeprobe/mining/curator_tiers.py:388-394`
- What: The docstring states: "The function invokes `call_claude` before returning any tier assignment (even when `use_llm=False` falls back to the heuristic result) — this keeps the control flow ZFC-compliant." This claim is incorrect. When `use_llm=False` or `not llm_available()`, the function returns `dict(heuristic)` at line 410 without calling `call_claude`. The `use_llm=False` code path is a documented offline fallback — which is legitimate — but the docstring's ZFC compliance justification is false for that path.
- Why it matters: CLAUDE.md is authoritative on ZFC compliance. Incorrect compliance claims in docstrings mislead future reviewers and the ZFC lint tool into treating a heuristic-only path as compliant. If the `--no-llm` flag triggers `use_llm=False`, the emitted tiers are purely heuristic (BFS graph distance) with no model involvement.
- Suggested fix: Fix the docstring: remove the parenthetical `"(even when use_llm=False..."`. Add a note that `use_llm=False` is a known offline fallback, equivalent to the pre-existing known violations in CLAUDE.md. Optionally add it to CLAUDE.md's known violations table.

## Simplification opportunities

### [infra-mcp-capabilities] Jinja2 `Environment` rebuilt on every `render()` call

- Current: `src/codeprobe/preambles/templates/__init__.py:55` — `_build_environment()` is called inside `render()`, creating a new `Environment`, `FileSystemLoader`, and compiling templates on every call.
- Proposal: Cache the environment at module level (a module-level `_ENV: Environment | None = None` or use `functools.lru_cache(maxsize=1)` on `_build_environment`). The environment is stateless (fixed `_TEMPLATES_DIR`, `StrictUndefined`, same flags every time).
- Blast radius: Minimal. The environment is immutable after construction; tests call `render()` multiple times and would benefit from the cache. The only risk is if a test monkeypatches `_TEMPLATES_DIR`, but none do.

### [r5-trace-sqlite] `run_eval` has a manual `trace_overflow` string validation that duplicates Click's `choice`

- Current: `src/codeprobe/cli/run_cmd.py:455-463` — manual `if trace_overflow not in ("fail", "truncate"):` check with `raise SystemExit(2)` inside `run_eval`, even though `--trace-overflow` is declared as `click.Choice(["fail", "truncate"])` in the CLI layer.
- Proposal: Remove the redundant guard in `run_eval`. `click.Choice` already rejects invalid values before `run_eval` is reached from the CLI. The guard is only relevant when `run_eval` is called programmatically (e.g., from tests), and the tests use valid values. If defensive protection of the programmatic API is desired, raise `ValueError` (not `SystemExit`) at the `run_eval` boundary.
- Blast radius: Tiny — the existing `test_run_trace_db.py` uses valid values. Removing `SystemExit(2)` from a library function improves testability.

### [r15-incremental-mining] `RetryTracker` mutates a dataclass — inconsistent with project immutability convention

- Current: `src/codeprobe/mining/retry.py:47-57` — `@dataclass` with `attempts: int = 0`, `exhausted: int = 0`, `history: list[str]` mutated in-place via `record_attempt()` / `record_exhaustion()`.
- Proposal: This is explicitly a stateful counter — immutable dataclass (`frozen=True`) would not work here. The naming and usage are clear. However, the `history` list is a `@dataclass` field that is mutated, which is inconsistent with the project's immutability convention for dataclasses. Consider using a plain class with `__init__` and `__repr__` instead of `@dataclass` to signal "this is stateful." Low priority.
- Blast radius: Tests would need minor updates.

### [mining-writer-parity] `writer.py` is 1658 lines — well above the 800-line project ceiling

- Current: `src/codeprobe/mining/writer.py` — 1658 lines. The file now contains inline oracle scripts (`_ORACLE_PY`, `_ORACLE_STRUCTURED`, large shell-script strings), instruction assembly, structured-retrieval schema, MCP variant logic, and weighted checklist builders.

The structured retrieval oracle alone is ~200 lines of inline Python string. Moving it to `src/codeprobe/mining/oracles/oracle_structured_retrieval.py.tmpl` or similar would bring `writer.py` back under 1000 lines and make the oracles independently testable.
- Blast radius: Medium — `writer.py` imports and embeds these strings; callers that inspect the oracle content in tests would need updating. But the tests in `test_oracle_structured_retrieval.py` already test the oracle logic separately, suggesting the extraction is already partially anticipated.

## Test quality notes

- `tests/mcp/test_capabilities.py:test_no_hardcoded_tool_names_in_capability_layer` — the test checks two specific tool names (`sg_nls_search`, `sg_deep_reason`) are absent from two specific files. This is a guardrail against a known past pattern, not a general hardcoded-name check. It passes trivially if different hardcoded names are introduced. Not a tautology, but narrow.
- `tests/trace/test_content_policy.py` — `test_deny_glob_matches_tool_output` uses `"*/etc/passwd*"` as the glob. This glob matches substrings of a path because of the leading `*`. The `fnmatch.fnmatch(redacted, glob)` call requires a *full-string* match, so `"cat /etc/passwd"` is correctly redacted but `"the file /etc/passwd is dangerous"` would also match `"*/etc/passwd*"`. Tests do not cover a case where a partial-path glob unexpectedly replaces benign output. Consider adding a test for false-positive glob behavior.
- `tests/cli/test_run_cmd_resolved_instruction.py:test_resolved_instruction_written_and_matches_agent_prompt` — the test asserts `instruction.resolved.md` contains the `instruction_text` string. This is correct but only verifies inclusion, not byte-exactness of the preamble resolution. The byte-exact assertion is against `adapter.run_calls[0][0]`. Good coverage overall.
