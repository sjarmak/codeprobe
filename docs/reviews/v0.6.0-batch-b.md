- One-paragraph overall impression

The batch is architecturally coherent: it introduces four named extension points (dependency_upgrade task type, cross-repo AST ranking, NarrativeAdapter, VCS/tracker adapters) plus two infrastructure layers (preamble rendering, LLM registry). ZFC compliance is generally strong тАФ almost every semantic decision is delegated to a model call or left to the caller. The auth-hygiene contract in r10 is particularly well-built. The main concerns are a silent AuthFailure swallow in `pr_context`, placeholder Bedrock ARNs shipped in the registry, a private-function coupling between `multi_repo` and `extractor`, and several abstractions that ship with only one implementation and no concrete second use case in scope. `multi_repo.py` is exactly at the 850-line limit and should be split before the next feature lands on top of it.

## Critical findings

### [r10] `AuthFailure` is silently swallowed in `GitLabAdapter.pr_context`

- File: `src/codeprobe/mining/vcs/gitlab.py:160тАУ165`
- What: `AuthFailure` is a subclass of `RuntimeError`. The `except RuntimeError: commits = []` guard around the commits sub-call silently demotes a 401/403 to an empty list. The inline comment ("if this sub-call fails on auth we still fail loud because it shares `_check_auth`") is incorrect тАФ `_check_auth` raises `AuthFailure(RuntimeError)`, which the immediately surrounding `except RuntimeError` then catches and discards.
- Why: This means a revoked token or a permissions downgrade on the commits endpoint produces a silently incomplete task context rather than a loud failure, violating the project's fail-loud auth contract stated in the module docstring.
- Fix: Catch only `RuntimeError` errors that are **not** `AuthFailure`, or catch a narrower base exception:

```python
# BAD (current) тАФ catches AuthFailure because it is a RuntimeError subclass
try:
    commits = self._get(...)
except RuntimeError:
    commits = []

# GOOD тАФ re-raises AuthFailure, tolerates non-auth API errors
try:
    commits = self._get(...)
except AuthFailure:
    raise
except RuntimeError:
    commits = []
```

## High

### [r13] Bedrock ARNs contain placeholder account IDs тАФ will silently send `000000000000` to AWS

- File: `src/codeprobe/llm/model_registry.yaml:19,28,37`
- What: All three Bedrock ARNs carry `000000000000` as the account ID (e.g. `arn:aws:bedrock:us-east-1:000000000000:inference-profile/...`). Bedrock inference-profile ARNs must contain the caller's real AWS account ID.
- Why: Any code path that calls `BedrockBackend.complete` will send a malformed ARN to AWS and receive an `AccessDeniedException`. This is not caught at startup тАФ it will surface only at runtime on the first actual call, mid-run.
- Fix: Either replace with a clearly invalid sentinel (e.g. `REPLACE_WITH_YOUR_AWS_ACCOUNT_ID`) that produces an obvious error on YAML load, or add a validation step in `BedrockBackend.resolve_model_id` that rejects ARNs containing `000000000000`. Document that users must substitute their account ID before enabling the Bedrock backend.

### [r13] Model ID divergence between `core/llm.py` and `model_registry.yaml` тАФ `opus` version mismatch

- File: `src/codeprobe/llm/model_registry.yaml:18` vs `src/codeprobe/core/llm.py:57`
- What: `model_registry.yaml` names the Opus model `claude-opus-4-7` (with date-less slug). `core/llm.py` uses `claude-opus-4-6-20250514`. These are different models. There are now two separate, divergent model-name tables in the same codebase that are not connected to each other.
- Why: Any caller using the new `AnthropicBackend` from `codeprobe.llm` will hit `claude-opus-4-7`, while any caller using `core/llm.py`'s `call_claude` path will use `claude-opus-4-6-20250514`. Silent behavioral divergence across eval runs; no test flags the inconsistency.
- Fix: Either make `core/llm.py` look up models through `ModelRegistry` (unifying them), or add a CI-level test that asserts the logical name keys in the registry map to the same models used by `core/llm.py`. At minimum, add a comment in both files linking them together.

### [r8] Private function import coupling: `multi_repo` reaches into `extractor._get_changed_files`

- File: `src/codeprobe/mining/multi_repo.py:37`
- What: `multi_repo.py` imports `_get_changed_files` (a leading-underscore private function) directly from `codeprobe.mining.extractor`. This is a layering violation: two modules at the same layer are tightly coupled through an undeclared internal.
- Why: Any refactor of `extractor._get_changed_files` (signature change, rename, extraction to a helper module) silently breaks `multi_repo`. The coupling is invisible to `extractor`'s maintainer because it is not exported in `__all__`.
- Fix: Promote `_get_changed_files` to a public function in `extractor.py` (or a shared `git_utils` helper), add it to `__all__`, and import it from there. This makes the dependency explicit and flags it as a stable interface.

### [r8] `multi_repo.py` is at the 800-line project ceiling тАФ next feature will push it over

- File: `src/codeprobe/mining/multi_repo.py` (850 lines)
- What: The file already exceeds the project's 800-line soft ceiling. It contains six distinct responsibilities: `SymbolResolver` protocol, `RipgrepResolver` implementation, `mine_tasks_multi` dispatcher, `_mine_callers` implementation, `discover_related_repos` with interactive prompt, and five manifest parsers.
- Why: Two planned but unimplemented families (`refactor`, `dependency`) will need to land in this file if it is not split first.
- Fix: Extract manifest parsing to `mining/manifest_parsers.py`, move `RipgrepResolver` + `SymbolResolver` to `mining/symbol_resolver.py`, and leave `mine_tasks_multi` + `discover_related_repos` as the public API of `multi_repo.py`. Target <400 lines each.

## Medium

### [r10] `pr_context` commits sub-call has no timeout override

- File: `src/codeprobe/mining/vcs/gitlab.py:161тАУ163`
- What: The commits sub-call in `pr_context` routes through `_get` тЖТ `_http_get` which uses the default `timeout=15.0`. This is the correct value, but it is not obvious to readers that a second network round-trip is embedded inside `pr_context`. The outer caller in the mining pipeline has no way to bound the combined latency.
- Suggestion: Document the two-call nature of `pr_context` in the docstring with the per-call timeout, so callers can set appropriate overall timeouts.

### [r13] `_DEFAULT` registry singleton has no thread-safety guard

- File: `src/codeprobe/llm/__init__.py:161тАУ169`
- What: `get_registry()` uses a `global _DEFAULT` with a lazy `if _DEFAULT is None` check. This is a classic double-checked locking gap: two threads can both see `_DEFAULT is None` simultaneously, construct two `ModelRegistry` instances, and the second overwrites the first without any lock.
- Mitigation: In CPython with the GIL, the race is mostly theoretical for module-level singletons. However, the project's YAML reload mechanism (`ModelRegistry.reload()`) is mutable state that could also race. If multi-threaded parallel runs are ever introduced (they are planned per the product direction notes), this will need a `threading.Lock`.

### [r3] `no_llm=True` bypass is a production-reachable ZFC escape hatch via `--no-llm` CLI flag

- File: `src/codeprobe/mining/dependency_upgrade.py:307`, `src/codeprobe/cli/__init__.py:281`
- What: `generate_tasks(..., no_llm=True)` skips model classification and accepts all structurally filtered PRs as dependency upgrades. The `--no-llm` flag is a production CLI option (not test-only). The module docstring notes this as "offline/deterministic tests only," but there is no runtime warning when a user runs `codeprobe mine --no-llm` with `dependency_upgrade` task type тАФ they silently get unclassified tasks.
- Suggestion: Add a `logger.warning(...)` call when `no_llm=True` is used with the `dependency_upgrade` generator in production, matching the warning emitted in other `--no-llm` paths.

### [r9] `PRAdapter` blindly takes `items[0]` when a SHA search returns multiple PRs

- File: `src/codeprobe/mining/adapters/pr.py:71`
- What: `gh pr list --search <sha>` can return multiple results when a SHA appears in multiple PRs (e.g., cherry-picks). The adapter silently takes the first match.
- Why: In cross-repo setups where commits are cherry-picked into release branches, this silently picks the wrong PR body, feeding incorrect narrative context to the instruction generator.
- Fix: If `len(items) > 1`, log a debug/info message with the PR numbers so callers can see the ambiguity.

### [r10] `_http_get` is duplicated verbatim between `gitlab.py` and `jira.py`

- File: `src/codeprobe/mining/vcs/gitlab.py:36тАУ59`, `src/codeprobe/mining/trackers/jira.py:39тАУ57`
- What: The two `_http_get` implementations are nearly identical stdlib GET wrappers. The jira version's docstring explicitly acknowledges this ("mirrored shape with gitlab._http_get for test symmetry"), but the body is copy-pasted rather than shared.
- Why: The gitlab version is missing the `(UnicodeDecodeError, json.JSONDecodeError)` catch in the JSON parse step that the jira version has (diff confirmed). This means a non-UTF-8 GitLab API error body will raise an unhandled `UnicodeDecodeError` rather than returning a safe fallback.
- Fix: Extract a shared `_stdlib_get(url, headers, timeout)` function into `codeprobe/mining/vcs/base.py` (or a new `codeprobe/net/http_utils.py`) and have both adapters import it. Apply the jira version's `UnicodeDecodeError` handling to the extracted shared function.

## Simplification opportunities

### [r9] `NarrativeAdapter` Protocol has three real implementations; justified тАФ but `build_adapter` in `adapters/__init__.py` is a manual dispatch dict

- Current: `build_adapter(name)` uses three `if name == "..."` branches.

### [r8] `RipgrepResolver` + `SymbolResolver` Protocol have a single concrete implementation

- Current: `SymbolResolver` is a `@runtime_checkable Protocol` with one production implementation (`RipgrepResolver`). The Protocol exists for test substitutability.
- Assessment: Justified by the existing test fixtures (which pass `FakeResolver` stubs). No simplification needed тАФ keep as is. ### [r13] Five backend classes share identical `__init__`, `resolve_model_id` validation boilerplate

- Current: Each of `AnthropicBackend`, `BedrockBackend`, `VertexBackend`, `AzureOpenAIBackend`, `OpenAICompatBackend` duplicates the same `__init__(registry=None)` pattern and registry-fetch guard.
- Proposal: Extract a `_BaseBackend` mixin with `__init__` and a `_resolve_str_id` helper. Saves ~10 lines per backend (50 lines total), reduces copy-paste bugs. Blast radius: five backend files.

### [r12] `render_preamble` is not yet wired into any CLI command тАФ ships as library-only API

- Current: `preambles/generator.py:render_preamble` has no CLI entry point. It is callable only programmatically. The `--preamble` CLI flag resolves built-in `.md` names via `get_builtin`, not via `render_preamble`.
- Assessment: If the intention is to add `codeprobe preambles render --custom ...` in a follow-up, that is fine. If `render_preamble` was meant to be reachable from the current release, the wiring is missing. Track as a follow-up rather than blocking this PR.

### [r13] `codeprobe.llm` and `codeprobe.core.llm` are parallel LLM stacks with no shared code

- Current: `src/codeprobe/core/llm.py` handles all existing internal judgment calls (mining, assess). `src/codeprobe/llm/` is the new multi-backend registry. They share no code, use different model name spaces, and are imported by different call sites.
- Assessment: The intent appears to be a future migration path. The risk is that the two stacks will drift further apart (already: different opus versions, different naming conventions). A concrete migration plan or a code comment explaining the intended relationship would reduce the chance of both stacks being maintained independently forever.

## Cross-cutting themes

1. **Parallel LLM stacks** (`core/llm.py` vs `codeprobe/llm/`): The two stacks co-exist without a migration plan. The version divergence on the Opus model and the different logical-name schemes (`"haiku"` vs `"haiku-4.5"`) are early signs of drift. A follow-up bead should either migrate `core/llm.py` to use the new registry or document a phased sunset timeline.

2. **Protocol-without-second-implementation pattern**: `NarrativeAdapter` (r9), `SymbolResolver` / `VCSAdapter` / `IssueAdapter` (r8/r10) are all Protocols with exactly one production implementation each. This is not a violation of YAGNI in isolation тАФ test-driven design needs the seam. The risk is that future implementers add a second adapter and discover the Protocol does not cleanly capture what they need, leading to a breaking interface change. None of these Protocols have a stabilization marker (e.g., a `# STABLE` comment or version note) to set expectations.

3. **Missing pagination on enterprise adapters**: `GitLabAdapter.list_merges` is hard-bounded by `per_page=limit`. For enterprise customers mining large projects this will silently truncate. A generator-based pagination approach (yielding pages, stopping when no `X-Next-Page` header) would fit the existing `Iterator[MergeRequest]` return type without breaking callers.

4. **The `_get_changed_files` private-import pattern** may be the tip of a broader coupling iceberg between `multi_repo.py` and `extractor.py`. Both modules are in `codeprobe.mining` but are meant to be separate responsibilities (single-repo vs multi-repo mining). A `codeprobe.mining.git_utils` module collecting shared subprocess primitives (`_get_changed_files`, `_recent_commits`, `_head_sha`) would make the dependency explicit and public.
