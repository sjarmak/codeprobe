## Critical findings

### [r14] `hmac.new` is not a valid Python stdlib call — HMAC is silently constructed as a module attribute, not a function

- File: `src/codeprobe/snapshot/redact.py:289`, `redact.py:382`
- What: The code calls `hmac.new(key.encode("utf-8"), body, hashlib.sha256).hexdigest()`. In the Python standard library, the constructor is `hmac.new(key, msg, digestmod)` — this happens to exist as `hmac.new` in CPython but it is an **undocumented alias** that was formally deprecated in Python 3.4 and is listed for removal. The correct modern call is `hmac.HMAC(key, body, digestmod=hashlib.sha256)` or the shorthand `hmac.digest(key, body, hashlib.sha256)`. On environments that strip deprecated shims (some PyPy builds, Jython, future CPython), this will raise `AttributeError` at HMAC-signing time, causing `snapshot create` to crash for any user who passes `--signing-key`.
- Why this is critical: The HMAC path is the tamper-detection guarantee for enterprise snapshots. A crash here silently falls through to unsigned mode or exits non-zero in a context where the caller expects a signed manifest.
- Fix: Replace both occurrences with `hmac.new(key.encode("utf-8"), body, hashlib.sha256)` → use `hmac.digest(key.encode("utf-8"), body, hashlib.sha256)` (one-shot, returns bytes) or `hmac.HMAC(key.encode("utf-8"), body, digestmod=hashlib.sha256)` (object form). The existing tests happen to pass because CPython still exposes `hmac.new`, so this won't surface in CI until the alias is removed.

### [r16] `--offline` flag does not prevent network I/O — it is documentation intent only

- File: `src/codeprobe/cli/check_infra.py` (the entire `offline` subcommand), `src/codeprobe/net/__init__.py`
- What: R16's PRD promise ("--offline flag asserts no outbound traffic besides configured LLM/MCP endpoints") is implemented exclusively as a TTL preflight advisory. There is no `--offline` guard wired into the actual run path, no socket interception, no environment variable the agent harness reads to refuse outbound calls. The `offline_cmd` in `check_infra.py` is a standalone diagnostic that checks credential expiry before a run — but the `run` command itself has no corresponding `--offline` flag (grep of `cli/__init__.py` confirms `offline` does not appear as a `run` CLI option). An enterprise customer running `codeprobe run --offline` would get a `UsageError: no such option`. The acceptance criterion ("run in airgapped test VM completes end-to-end with stubbed endpoints") is structurally unachievable with the current implementation because the flag does not exist on `run`.
- Why this is critical: The entire R16 rationale is regulated/airgapped deployment. Customers who block on this feature for procurement will assume the flag works, deploy in a VPC, and discover mid-run that it does nothing.
- Fix: At minimum, add `--offline` to the `run` command and have it (a) invoke `check-infra offline` as a preflight and (b) set an env var (`CODEPROBE_OFFLINE=1`) that any network-touching module checks before initiating I/O. Mark this as explicitly scoped to the preflight-only model in docs so the gap is visible.

## High

### [r20] Dead `elif` branch in `signature_from_task` silently skips `dual` task files

- File: `src/codeprobe/mining/refresh.py:255`
- What: The `elif` clause at line 255 is unreachable. It reads:
  ```python
  elif v.verification_mode == "dual" and v.oracle_type == "file_list":
  ```
  But the `if` block on line 252 already consumed every case where `v.oracle_type` is truthy (`if v.oracle_type:`). When `v.verification_mode == "dual"` and `v.oracle_type == "file_list"`, the `if` branch on line 252 matches first (because `"file_list"` is truthy), so the `elif` never fires. The net effect: `dual` tasks whose `oracle_type` is `"file_list"` will produce the correct result from the `if` branch, but the code misleadingly implies there is a second distinct path. If `oracle_type` were ever `""` or `None` (e.g., a legacy dual task emitted by an older writer), the `elif` silently falls through to the `else` branch and returns `oracle_type = "unknown"` with an empty file set. This causes a false-positive structural mismatch on refresh.
- Fix: Remove the dead `elif` and add explicit handling for legacy dual tasks with empty `oracle_type`:
  ```python
  if v.oracle_type:
      oracle_type = v.oracle_type
      raw_files = v.oracle_answer
  elif v.verification_mode == "dual":
      # Legacy dual task with no oracle_type set — treat as file_list
      oracle_type = "file_list"
      raw_files = v.oracle_answer
  else:
      ...
  ```

### [r19] No CSV formula injection protection in `sheets.py` and `sigma.py`

- File: `src/codeprobe/snapshot/exporters/sheets.py:27-40`, `src/codeprobe/snapshot/exporters/sigma.py:69-81`
- What: Neither exporter sanitizes values for CSV/spreadsheet formula injection. A task entry whose `task_id` or any string field begins with `=`, `+`, `-`, or `@` will be interpreted by Google Sheets, LibreOffice Calc, and Excel as a formula when pasted. The `_cell()` function in `sheets.py` and `_stringify()` in `sigma.py` both pass through raw strings without prefix-checking. Since `aggregate.json` entries come from agent output — external, untrusted data — this is a real injection surface. Example: a task named `=HYPERLINK("http://attacker/x","click")` would render as a live hyperlink in Sheets.
- Why it matters: The explicit PRD goal is one-click-paste into Sheets; the user flow is copy-entire-file-paste, which maximizes the injection blast radius.
- Fix: In both `_cell()` and `_stringify()`, prefix-check the rendered string:
  ```python
  _FORMULA_PREFIXES = ("=", "+", "-", "@", "\t", "\r")
  if rendered and rendered[0] in _FORMULA_PREFIXES:
      rendered = "'" + rendered  # TSV-safe prefix quoting
  ```
  For sigma's CSV writer, use `csv.QUOTE_ALL` or wrap the sanitized value. Add a test with a fixture entry whose `task_id` starts with `=HYPERLINK(...)` and assert it does not appear unquoted in the output.

### [r18] `_verify_file_hashes` always appends redacted-body candidates as offenders

- File: `src/codeprobe/snapshot/verify.py:162-172`
- What: In `contents` or `secrets` mode, the on-disk body in `files/<rel>` has been transformed by the scanner — secrets are replaced with `[REDACTED:rule_id]` markers. The hash of the redacted body will never equal the source hash recorded in `SNAPSHOT.json`. The verification logic at line 168 computes `actual != expected` and then at line 172 appends to `offenders` with a comment acknowledging the legitimate mismatch, but the condition has no guard — it always adds the path. The comment at line 174 says "Any remaining offenders indicate tampering; if the hash diff is solely from redaction the test can use a MockScanner with no hits so redaction is a byte-for-byte copy." This means the `verify` command will **always fail** on a non-trivial `contents` snapshot where any file was actually redacted. The tamper-detection contract for `contents` mode is broken: the command flags legitimate snapshots as tampered.
- Fix: Store a `redacted_body_sha256` per file entry in the manifest (alongside the source `sha256`) and verify the redacted-body hash against that value. If the redacted hash is absent, fall back to comparing the source hash (catches cases where redaction was a no-op for that file).

### [r14] `contents` mode bypasses the canary gate

- File: `src/codeprobe/snapshot/redact.py:174`, `src/codeprobe/cli/snapshot_cmd.py:172-206`
- What: The canary gate (`require_pass_or_raise`) is only enforced when `mode == "secrets"`. The `contents` mode writes scanner-redacted file bodies without running the canary gate at all. Both modes copy actual source file content; the only difference is that `secrets` requires the canary proof. A user who wants to share source content can simply use `--redact=contents --allow-source-in-export` and entirely bypass the "does the scanner actually work?" verification. The PRD's capability matrix section says the canary gate is a pre-publish requirement; the current code makes it skippable by mode selection.
- Fix: Require the canary gate for `contents` mode as well, or explicitly document in the capability matrix that `contents` mode has no scanner-effectiveness guarantee (not just a documentation fix — the capability matrix claimed equivalence).

### [r17] Hardcoded `0.5` pass threshold in `step2_scope_correct.sh` diverges from `PASS_THRESHOLD`

- File: `src/codeprobe/mining/org_scale.py:80`
- What: The shell script embedded in `CHANGE_SCOPE_CHECKPOINT_SCRIPTS` uses the literal `0.5` as the F1 pass threshold: `awk -v s="$SCORE" 'BEGIN { exit (s+0 >= 0.5) ? 0 : 1 }'`. The canonical project-wide pass threshold lives in `src/codeprobe/analysis/stats.py:PASS_THRESHOLD = 0.5`. While they coincidentally agree today, the shell string is a baked-in constant that will not track a future `PASS_THRESHOLD` change. Because the script is emitted at mine time and persisted to disk, a repo using `PASS_THRESHOLD = 0.6` would have its checkpoint scripts disagreeing with the runtime scorer. This is the kind of silent drift that INV1 is designed to prevent.
- Fix: Either (a) generate the script dynamically, substituting `PASS_THRESHOLD` at write time so the emitted value is pinned to the mine-time value, or (b) have the script read a threshold from a config file dropped alongside it by `write_task_dir`.

## Medium

### [r18] Snapshot `SNAPSHOT.json` rewritten twice — r14 attestation covered by a second write that could diverge

- File: `src/codeprobe/snapshot/create.py:290-305`
- What: `create_snapshot` calls `redact(...)` which writes `SNAPSHOT.json`, then calls `write_extended_manifest(...)` which overwrites `SNAPSHOT.json` with R18 fields appended. The second write uses `manifest_to_json_dict` which calls `ext.base.to_dict()` — this re-serializes the r14 keys. If `SnapshotManifest.to_dict()` ever produces different output from what `redact()` wrote (e.g., key ordering, float precision), the `body_sha256` in the attestation will be stale because it was computed over the first write's payload. The attestation signature is therefore only valid if `_canonical_body_bytes` (which is also called by `verify_snapshot`) recomputes from the same in-memory `SnapshotManifest` object. This is currently true — `_canonical_body_bytes` ignores the extended fields — but the contract is implicit and fragile.
- Fix: Remove the first write entirely. Have `redact()` return the in-memory `SnapshotManifest` without writing to disk, and let `create_snapshot` call `write_extended_manifest` once with the fully-assembled manifest. This makes the write-once contract explicit.

### [r18] `_collect_issue_trackers` returns hardcoded version strings

- File: `src/codeprobe/snapshot/manifest.py:143-146`
- What: The `_collect_issue_trackers()` function returns `[{"name": "jira", "api_version": "v3"}, {"name": "github", "api_version": "v3"}, {"name": "gitlab", "api_version": "v4"}]` as a literal. The PRD purpose of this block is to record the "API versions present at mine/run time" so a snapshot reproduced six months later can detect why it no longer reproduces. These hardcoded strings do not actually reflect the runtime-configured tracker or its version — they are always the same regardless of what adapters are configured. A customer not using Jira will still see `jira: v3` in every snapshot.
- Fix: Either source these from the configured adapters (checking which VCS/tracker adapters are registered at mine time) or honestly document that these are build-time constants, not runtime observations.

### [r14] `shutil` is imported but unused — kept alive via suppression hack

- File: `src/codeprobe/snapshot/redact.py:36`, `redact.py:420-423`

This is a speculative YAGNI abstraction that suppresses the linter warning rather than removing the import. It violates the project's import hygiene and is exactly the kind of "just in case" pattern the architecture rules prohibit.

### [r11] `load_canary_proof` in `canary.py` does not validate `passed=True` before returning

## Simplification opportunities

1. **r14/r18 double-write of `SNAPSHOT.json`**: `redact()` writes the manifest, then `write_extended_manifest()` overwrites it. This should be a single write (see Medium finding above). Removing the intermediate write also eliminates the implicit coupling between the two serialization paths.

2. **r19 three-exporter glue duplication**: Each of `datadog.py`, `sigma.py`, and `sheets.py` independently calls `load_manifest(snapshot_dir)` and `load_entries(snapshot_dir)` with identical error semantics. The `_common.py` module already centralises loading; a thin `ExportContext` dataclass returned by a single `load_export_context(snapshot_dir)` call would remove three copies of the same two-line load preamble and make it impossible for a new exporter to forget to load the manifest.

3. **r14 `canary.py` double-check logic in `CanaryGate.prove()`**: Lines 70-85 implement the same "does any finding overlap the canary span?" check twice — once inline (lines 70-73) and once in a fallback block (lines 78-85). The `_canary_overlaps` helper at line 106 is a third restatement of the same logic. Consolidate to a single call of `_canary_overlaps` per finding.

4. **r11 calibration scaffolding**: All live code paths in `calibration/gate.py` and `calibration/profile.py` are reachable and correct. The only dead surface is the partner-gated acceptance criteria (≥100 tasks, ≥3 non-OSS repos) — these are enforced in code but will always fail until partner data lands. This is intentional and correctly documented; no simplification needed. However, `calibrate_cmd.py` exposes a `--allow-unvalidated-calibration` flag that was not visible in the CLI `__init__.py` registration — confirm it is properly gated and not exposed as an unlisted opt-in bypass.

5. **r20 `mine_cmd.py` duplicated refresh call sites**: `accept_structural_change` is passed through two separate code paths in `mine_cmd.py` (lines 1998-2013 and lines 2667-2763). If the refresh logic grows, both sites will need updating. Extract a `_run_refresh(task_dir, new_task, new_commit, accept_structural_change)` helper.

6. **r18 `_write_empty_summary` is called unconditionally after `_populate_summary`**: `_populate_summary` calls `_write_empty_summary` as its final step anyway — a second call from `create_snapshot` is a no-op. Remove the redundant call pattern by documenting that `_populate_summary` guarantees all four files exist after it returns.

## Security review (spotlight on r14/r17/r18/r19)

**Path traversal surface (r18):** `preflight_symlink_containment` correctly addresses the classic tarfile-style path traversal. One remaining surface: `_iter_trial_dirs` iterates `experiment_dir/<config>/<task_id>/` using `sorted(experiment_dir.iterdir())`. A malicious `experiment_dir` containing a directory named `../../../etc` would have its name passed through `config_dir.name` (which returns the basename only, not the full path), so `shutil.copytree(source, dest)` uses an untrusted `config` string in `layout.export_traces_dir / config / task_id`. If `config` were `../../../etc`, the destination would escape the snapshot directory. The symlink preflight does not catch this because it is not a symlink. This path is only reachable for callers who pass an attacker-controlled `experiment_dir` — the CLI validates `exists=True, dir_okay=True` but does not sanitize subdirectory names. Mitigation: strip or validate `config` and `task_id` to alphanumeric/dash/underscore before constructing destination paths.

**CSV/formula injection (r19):** Confirmed unfixed (see High finding). Both `sheets.py` and `sigma.py` are affected. `datadog.py` is safe because it produces JSON, not CSV/TSV.

**Secret redaction gaps (r14):** The `PatternScanner` patterns cover major known token formats. The explicit documentation that `secrets` mode is "not sufficient for external sharing" is correct. The `hashes-only` default eliminates the class of "undetected secret in body" entirely. The canary gate for `contents` mode bypass is a gap (see High finding).

**Shell injection in checkpoint scripts (r17):** The verifier scripts are generated strings written by codeprobe at mine time from `CHANGE_SCOPE_CHECKPOINT_SCRIPTS` — a module-level constant, not user input. The `TASK_DIR` and `SCORE` variables are constructed from known filesystem paths and `awk` output. No user-controlled data flows into the shell command expansion. Shell injection risk is low.

**airgap browse.html XSS surface (r19):** The HTML generation escapes all user-controlled data through `html.escape`. The JSON island uses `_safe_json_island` which escapes `</` sequences. The inline JS is static and never constructs `innerHTML` from the data island — it only manipulates DOM node ordering via `tbody.appendChild`. No `eval`, no `document.write`. The XSS surface is considered closed.

One residual concern: the `source` field from `SNAPSHOT.json` is a filesystem path (e.g., `/home/user/experiments`). It is escaped via `html.escape` before HTML embedding and passed raw into the JSON island. If an attacker controls the `source` field in the manifest (by providing a crafted `SNAPSHOT.json`), the path would appear unescaped in the JSON island but escaped in the HTML meta block. Since the JSON island is `type="application/json"` it is not parsed as HTML, so this is acceptable.

## Cross-cutting themes

1. **Double-write anti-pattern:** Both r14→r18 (`SNAPSHOT.json` written twice) and r20's refreshed-task write-and-then-serialize pattern rely on in-memory state being faithfully reproduced on second serialization. The architectural fix is write-once by composing all layers before committing to disk.

2. **Offline flag completeness:** R16's implementation correctly handles credential TTL preflight but does not extend to the run path. This is a consistent architectural gap — the `check-infra offline` subcommand is treated as a standalone diagnostic rather than a gate wired into the execution pipeline.

3. **Threshold constant proliferation:** `PASS_THRESHOLD` (0.5), `CHURN_THRESHOLD` (0.20), correlation threshold (0.6), calibration holdout minimums (100 tasks, 3 repos), and the checkpoint `awk >= 0.5` test are all defined in different files. None are cross-referenced. A single `src/codeprobe/constants.py` exporting the numeric policy constants would make these auditable in one place and prevent the `step2_scope_correct.sh` divergence class.

4. **`shutil` YAGNI import:** `redact.py` keeps a dead import alive via a suppressor assignment. This pattern is a minor violation of the codebase's YAGNI and import hygiene rules and sets a precedent for future "just in case" imports.
